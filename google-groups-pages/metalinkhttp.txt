<h1>metalinkhttp</h1>
The latest version of the Internet Draft of the Metalink Download Description is at <a href="http://tools.ietf.org/html/draft-bryan-metalinkhttp">http://tools.ietf.org/html/draft-bryan-metalinkhttp</a><br>(Incremental updates are at <a href="http://metalinks.svn.sourceforge.net/viewvc/metalinks/internetdraft/">http://metalinks.svn.sourceforge.net/viewvc/metalinks/internetdraft/</a> )<br><br>Example:<br><pre>   Link: &lt;http://www2.example.com/example.ext&gt;; rel="duplicate"<br>   Link: &lt;ftp://ftp.example.com/example.ext&gt;; rel="duplicate"<br>   Link: &lt;http://example.com/example.ext.torrent&gt;; rel="describedby";<br>   type="application/x-bittorrent"<br>   Link: &lt;http://example.com/example.ext.metalink&gt;; rel="describedby";<br>   type="application/metalink4+xml"<br>   Link: &lt;http://example.com/example.ext.asc&gt;; rel="describedby";<br>   type="application/pgp-signature"<br>   Digest: SHA=thvDyvhfIqlvFe+A9MYgxAfm1q5=<br></pre><br>Implementations:<br><ul><li><a href="http://metalinks.svn.sourceforge.net/viewvc/metalinks/checker/">metalink-checker</a><br></li><li><a href="http://metalinks.svn.sourceforge.net/viewvc/metalinks/webconvert/">Convert Metalink 3.0 to Apache directives for Metalink/HTTP</a><br></li></ul><br>Known issues:<br><br><ul><li>Use of Link header to describe Mirrors.&nbsp; Only send a few mirrors with Link header, or only send them if Want-Digest is used?&nbsp; Some organizations have many mirrors.</li><li>A way to differentiate between mirrors that have synchronized ETags and those that don't.</li><li>Do we want a way to show that whole directories are mirrored, instead of individual files?</li><li>Will we use Metalink/XML for partial file checksums? This is an important feature &amp; that would add XML dependency to apps.
</li><li>Do we need an official MIME type for .torrent files or allow "application/x-bittorrent"?
</li></ul>This draft compared to IETF ID Metalink/XML specification:<br><br><ul><li>(+) Reuses existing HTTP standards without defining anything new besides a Link Relation Type.&nbsp; It's more of a collection/&nbsp; coordinated feature set.<span><span></span></span></li><li><span><span>(?)&nbsp; The existing standards don't seem to be widely implemented.<br></span></span></li><li>(+) No XML dependency.</li><li>(+)&nbsp; Existing Metalink/XML clients can be easily converted to support this as well.</li><li>(+) Coordination of mirror servers is preferred, but not required.
Coordination may be difficult or impossible unless you are in control
of all servers on the mirror network.</li><li>(-?)&nbsp; Tied to HTTP, not as generic.&nbsp; FTP/P2P clients won't be using it unless they also support HTTP, unlike Metalink/XML.<br></li><li>(-) Requires software or configuration changes to originating server.</li><li>(-) Also, Metalink/XML files are easily mirrored on all servers. Even if usage in that case is not as transparent, it still gives access to users at all mirrors (FTP included) to all download information with no changes needed to the server.</li><li>(-) Not portable/archivable/emailable.&nbsp; Metalink/XML is used to import/export transfer queues.&nbsp; Not as easy for search engines to index?</li><li>(-) No way to show mirror geographical location (yet).</li><li>(-) Not as rich metadata.</li><li>(-) Not able to add multiple files to a download queue or create directory structure.<br></li></ul><br><wbr>
